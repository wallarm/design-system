# Tour code recipes

Copy-paste starting points for the three shapes the gate (SKILL.md step 1) routes to.
All three follow the `Tour.llm.md` contract: **config-driven** (never compose
`TourContent`/`TourFooter`/etc.), `target` is **a function returning the element**,
analytics ride the step `actions`. Everything imports from the package root.

> Prototype stubs: `localStorage` for the "seen" flag and `console.log` for analytics
> are placeholders — engineering wires real persistence / telemetry for production.
> Don't let that stop you shipping the prototype.

## Recipe A — on-demand guided coachmark tour (the default multi-step shape)

A welcome dialog → anchored coachmarks → a finish dialog, launched from a button,
shown once. Use for a genuinely unfamiliar multi-step area.

```tsx
import { useRef } from 'react'
import { Button, Tour, useTour, type TourStepDetails } from '@wallarm-org/design-system'

const SEEN_KEY = 'tour:dashboard-v1'

export function DashboardWithTour() {
  const filtersRef = useRef<HTMLButtonElement>(null)
  const exportRef = useRef<HTMLButtonElement>(null)

  const steps: TourStepDetails[] = [
    {
      id: 'welcome',
      type: 'dialog', // centered card, no target
      title: 'Your new dashboard',
      description: 'A faster way to see what matters. Take 30 seconds for the highlights.',
    },
    {
      id: 'filters',
      type: 'tooltip',
      target: () => filtersRef.current, // a function, never a selector string
      title: 'Focus the view',
      description: 'Narrow the dashboard by team, severity, or time range.',
      placement: 'bottom-start',
    },
    {
      id: 'export',
      type: 'tooltip',
      target: () => exportRef.current,
      title: 'Share what you find',
      description: 'Export the current view to CSV, or share a link with your team.',
      placement: 'bottom-end',
    },
    {
      id: 'done',
      type: 'dialog',
      title: "You're all set",
      description: 'Reopen the tour anytime from the help menu.',
      // actions are auto-generated by position; override only when you need to:
      actions: [{ label: 'Finish', action: 'dismiss' }],
    },
  ]

  const markSeen = () => localStorage.setItem(SEEN_KEY, '1') // prototype stub
  const tour = useTour({
    steps,
    closeOnInteractOutside: false,                 // a stray click shouldn't end a guided tour
    onFinish:  markSeen,
    onDismiss: markSeen,
    onSkip:    markSeen,                            // finish / dismiss / skip all retire it — a *skipped* tour must not re-show
    onStepChange: ({ stepId }) => console.log('tour:step', stepId), // analytics stub (single onFinish only — don't double-wire it)
  })

  return (
    <>
      <Button ref={filtersRef} variant="outline" color="neutral">Filters</Button>
      <Button ref={exportRef} variant="outline" color="neutral">Export</Button>

      {/* On-demand trigger — preferred over blanket autoStart. */}
      <Button variant="ghost" color="neutral" onClick={() => tour.start()}>
        Show me around
      </Button>

      <Tour tour={tour} />
    </>
  )
}
```

**First-run variant** (if it must auto-start the first time, keep it skippable):
gate on the flag — `const tour = useTour({ steps, autoStart: !localStorage.getItem(SEEN_KEY), closeOnInteractOutside: false, onFinish: markSeen, onDismiss: markSeen })`.

## Recipe B — single passive beacon (announce ONE feature, non-blocking)

The right size for "promote a new feature." A pulsing dot; the popover opens when the
user clicks the highlighted element. The rest of the UI stays live.

```tsx
import { useRef } from 'react'
import { Button, Tour, useTour, beaconStepEffect, type TourStepDetails } from '@wallarm-org/design-system'

export function NewExportBeacon() {
  const exportRef = useRef<HTMLButtonElement>(null)

  const steps: TourStepDetails[] = [
    {
      id: 'export-is-new',
      type: 'tooltip',
      target: () => exportRef.current,
      title: 'Export is here',
      description: 'Download the current view as CSV — find it anytime in the toolbar.',
      backdrop: false,            // non-blocking
      effect: beaconStepEffect(), // pulsing dot, opens on click (use { shape: 'circle' } for icon targets)
    },
  ]

  // Single step → the footer shows just "Got it".
  const tour = useTour({ steps, autoStart: true })

  return (
    <>
      <Button ref={exportRef} variant="outline" color="neutral">Export</Button>
      <Tour tour={tour} />
    </>
  )
}
```

## Recipe C — interactive "do-it" walkthrough (highest completion + activation)

The user actually performs each action before the tour advances. Prefer this on any
action-bearing step. A step with an `effect` **hides the nav buttons** — the prompt is
the *description*, and the user's action is the advance.

```tsx
import { useRef } from 'react'
import { Button, Input, Tour, useTour, waitForStepEvent, type TourStepDetails } from '@wallarm-org/design-system'

export function CreateRuleWalkthrough() {
  const createRef = useRef<HTMLButtonElement>(null)
  const nameRef = useRef<HTMLInputElement>(null)

  const steps: TourStepDetails[] = [
    {
      id: 'intro',
      type: 'dialog',
      title: 'Create your first rule',
      description: 'Three quick steps and your first rule is live.',
      actions: [{ label: 'Start', action: 'next' }],
    },
    {
      id: 'open-editor',
      type: 'tooltip',
      target: () => createRef.current,
      title: 'Open the editor',
      description: 'Click **New rule** to continue.',          // the prompt, not a Next button
      effect: args => waitForStepEvent('click', args),
    },
    {
      id: 'name-it',
      type: 'tooltip',
      target: () => nameRef.current,
      title: 'Name your rule',
      description: 'Type a name (3+ characters) to continue.',
      placement: 'bottom-start',
      effect: args =>
        waitForStepEvent('input', args, {
          predicate: el => (el as HTMLInputElement).value.trim().length >= 3,
          delay: 800,                                          // advance once typing settles (debounce, not a timer)
        }),
    },
    {
      id: 'done',
      type: 'dialog',
      title: 'Rule created',
      description: 'Nicely done — your rule is now protecting traffic.',
    },
  ]

  const tour = useTour({ steps, closeOnInteractOutside: false })

  return (
    <>
      <Button ref={createRef} variant="primary" color="brand">New rule</Button>
      <Input ref={nameRef} placeholder="Rule name" />
      <Tour tour={tour} />
    </>
  )
}
```

## Recipe D — deferred-mount (a dialog or form appears only after a click)
When the next element to anchor doesn't exist until the user acts (a dialog opens, a
config form renders), a plain `tooltip` step would target a **null** element. Gate the
step on the interaction that mounts the next target with `waitForStepEvent`, and attach
the next step's ref to the content that mounts on open.

```tsx
const steps: TourStepDetails[] = [
  {
    id: 'open-dialog',
    type: 'tooltip',
    target: () => addBtnRef.current,                  // exists now
    title: 'Start a new integration',
    description: 'Click **Add integration** to continue.', // the click IS the advance
    effect: args => waitForStepEvent('click', args),       // advances when the dialog opens
  },
  {
    id: 'pick-type',
    type: 'tooltip',
    target: () => typeSelectRef.current,              // lives INSIDE the dialog — mounts on open
    title: 'Choose a type',
    description: 'Pick the integration type to continue.',
    // ⚠️ The DS `Select` PORTALS its option menu (+ renders a HiddenSelect), so an
    // option click never bubbles to the wrapper ref. Listen on the inner native
    // <select> — same idiom as the Tour stories' checkbox→input target override.
    effect: args =>
      waitForStepEvent('change', { ...args, target: () => typeSelectRef.current?.querySelector('select') }),
  },
  {
    id: 'fill-form',
    type: 'tooltip',
    target: () => configFormRef.current,              // the config form — mounts after the pick
    title: 'Add your details',
    description: 'Fill the required fields, then save.',
  },
]
```
- Each target only needs to exist **by the time its step runs** — the prior step's
  interaction mounts it. Don't pre-check all refs up front.
- `closeOnInteractOutside: false` is essential here: clicking *into* the dialog must not
  read as "clicked outside, end the tour."
- If a target still isn't mounted when its step runs, the tour can't anchor — confirm
  the sequence in a preview (`play` fn) before hand-off.
- **Portaled controls (`Select`, menus, anything in a Portal):** an option click fires
  *inside the portal*, not on the wrapper element — so a `waitForStepEvent` listening on
  the wrapper ref never fires. Point its target at the **inner native control**
  (`…querySelector('select')` / `('input')`), or gate on a different signal (e.g. the
  controlled value changing). Confirm in a preview.

## Don't, in generated code
- Don't compose `TourContent` / `TourFooter` / `TourTitle` / `TourMedia` — they're
  internal and unexported; everything is the `steps` config. (Only `<TourClose>` is
  composable, for analytics.)
- Don't pass a CSS selector string to `target` — it's `() => element`.
- Don't hand-build Back/Next/Finish buttons or restyle them — they're auto-generated by
  step position; override labels only via a step's `actions` array.
- Don't add a Next button to an `effect` step (it's hidden), and don't try to
  auto-advance on a timer — Tour advances on interaction only.
- Don't add your own progress indicator — a step counter renders automatically for
  multi-step tours.
- Media: set `meta: { mediaSrc, mediaAlt }` on the step; don't build a media slot.

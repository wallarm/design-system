## Decision guides (consolidated ladders)

> The Backlog above parks **atomic** rules + their provenance. This section consolidates the recurring **pick-a-pattern** decisions into decision-ready form — the shape a PM / designer (or the future foundations skill) needs to actually *choose* a surface. First one written out below; the other ladders (messaging · filtering · collection-display · onboarding) get consolidated here on harvest.

### Overlay & layout surfaces — "show or do something on top of / beside the current view"

Two axes decide it. **Bias: the lightest surface that fits the content, and non-modal unless the task must block.**
- **Content weight** — a word → a hint → rich content → a focused task → an object's detail → a whole workflow.
- **Blocking** — does it **block** the page (modal), sit **beside** a still-usable page (non-modal), or **replace** the view (navigation)?

| What you need | Surface | Blocks the page? |
|---|---|---|
| A passive label / hint / shortcut on hover or focus | **`Tooltip`** | no — passive, no interaction |
| The full text of something **only when it's truncated** | **`OverflowTooltip`** | no |
| Rich / custom content or a few controls, opened by **click**, page stays live | **`Popover`** | no — non-modal, anchored |
| A short, focused decision or one-field task that should **interrupt** | **`Dialog`** | **yes** — modal, centered |
| An object's detail / preview / contextual edit **beside** the page | **`Drawer`** | **your choice** — modal (blocks; page still visible) *or* non-modal (`modal={false}`+`overlay={false}`, page interactive) |
| All of an object's detail, or a complex multi-section flow | **full page** | replaces the view (navigation) |

**Where the choice actually gets made (the distinguishers):**
- **Tooltip → Popover:** a passive *label* (hover, no interaction) vs *interactive / rich* content (click). The moment it's actionable or more than a brief hint, it's a `Popover`.
- **Popover → Dialog:** non-modal-anchored-to-a-trigger vs modal-centered-**blocking**. If the user must deal with it before continuing, it's a `Dialog`.
- **Dialog ↔ Drawer (the subtle one):** built on the same engine — **`Dialog` is literally `<Drawer>` centered + always-modal, minus resize.** `Dialog` = centered, modal, no resize, smaller (400 / 560 / 960). `Drawer` = right-edge side panel, **resizable** (640 / 960 / 1130, clamp 400–1130), and the **only overlay that can hold a full panel while keeping the page interactive** (non-modal). Pick `Drawer` when context beside the page matters or the user keeps working; `Dialog` when a small task should interrupt.
- **Drawer → full page:** a bounded side panel vs the full drill-down. Wants persistence, many sections, or a 2nd nesting level → a **page** (reached from the drawer's title link).

**Not on this ladder (don't reach here):** a command list → `DropdownMenu` · a transient outcome → `Toast` · an inline regional message → `Alert` · a page-wide standing notice → `Banner` · a guided multi-step intro → `Tour` · a form-field value from a list → `Select`. (Those belong to the messaging / actions / onboarding ladders.)

### Switcher surfaces — "move between peer views, scopes, or sections in place"

Three questions, in order. **Bias: underline `Tabs` is the default tab; reach for the segmented pill *deliberately*.**
1. **Does selecting it show a content panel the component owns?** No — it just emits a value, only re-presents the *same* content (table↔card, list↔board), or you render the result yourself → **`SegmentedControl`**. Yes → it's tabs-like.
2. **Different content *sections*, or *scoped views of one collection*?** Genuinely different sections (Overview / Activity / Settings) → **`Tabs`** (underline, primary level). The *same* collection under preset filters/scopes (All / Pending / Blocked; saved views — with counts + a saved-view dot + per-pill actions) → **`SegmentedTabs`** (pill skin).
3. **A *secondary* level under existing `Tabs`?** → **`SegmentedTabs`** — the distinct skin signals the second level (**two levels max**; a third wants real navigation).

| What you need | Surface | Owns a panel? |
|---|---|---|
| A view / mode toggle of the *same* content (table↔card, list↔board) or a compact value | **`SegmentedControl`** | no — emits a value |
| Primary, page / section content sections (Overview / Activity / Settings) | **`Tabs`** (underline) | yes |
| Scoped views / quick-filters of *one* collection (All / Pending / saved views) | **`SegmentedTabs`** (pill) | yes |
| A secondary tab level under primary `Tabs` (2 levels max) | **`SegmentedTabs`** (pill) | yes |
| Sibling sections all open at once | **`Accordion`** | n/a — not one-at-a-time |

**The two load-bearing distinguishers:** (a) **owns a panel?** — `SegmentedControl` emits a value and owns none (a *presentation* toggle of the same data, or you render the result); `Tabs`/`SegmentedTabs` own their panels. (b) **which content vs how it looks** — changing *which* content shows (a different scope/filter) is `SegmentedTabs`; changing *how the same content looks* (layout/format) is `SegmentedControl`.

**Off-ramps (not a switcher):** many / combinable / multi-select filters → `Select` / `FilterInput` · distinct top-level app areas → primary nav · hierarchy / location → `Breadcrumbs` · a sequence / wizard → a stepper.

# Tour — usage

> A step-by-step guided overlay that highlights real UI elements with anchored
> popovers (plus centered welcome / finish cards) to onboard users to a feature
> or workflow. **Config-driven** — you describe the tour as a `steps` array and
> render it once; you don't compose its parts. Built on Ark UI Tour. Interactive.

## Reach for it when
You want to walk a user through **a sequence** — a first-run intro to an
unfamiliar workflow, or guiding someone hands-on through a multi-step task they
haven't done before. The cue: when you'd otherwise hand-roll a "welcome modal +
a chain of positioned tooltips with a dimmed backdrop and Back/Next buttons" —
that's a Tour. Drive it with `useTour({ steps })` and render one `<Tour tour={tour} />`.

**Promoting a single new feature** can be a Tour job — but as **one** non-blocking
`beaconStepEffect` step, and only when it genuinely needs an in-context *point-at*.
Reach for the lightest tool first: a `Badge` "New", a `Tooltip`, or a `Banner`
covers most new-feature callouts; a multi-step promo march is the wrong size (boundaries below).

## Don't use it for
- **A single hover/focus hint on a control** → `Tooltip`. A tour is a *sequence*; one explanation isn't a tour.
- **Custom interactive UI anchored to a trigger** (a form, a mini-dashboard) → `Popover`. Tour steps are read-and-advance, not arbitrary UI.
- **A standing system/account notice** → `Banner`; **a transient action confirmation** → `Toast`; **a decision the user must make** → `Dialog`. (The messaging / overlay ladders — a tour is a one-time guided thing, not a notice or a blocker.)
- **A zero-data / first-use screen** → `EmptyState` (which may itself launch a tour).
- **Explaining the obvious, standard icons, or papering over a confusing screen** → don't. Tours don't fix bad UX and "explanations of standard icons waste users' time" (NN/g); fix the UI instead.
- **Feature-dumping** (showing every feature at once) → orient the whole tour to **one** user goal; a 7-step tour completes ~16% vs ~72% for 3 steps (Chameleon).
- **A "what's new" release note** → a time-boxed `Banner` / changelog, not a tour. Multi-step promotional tours at launch are usually the wrong tool (NN/g); a single coachmark is the right size.

## How it's driven (the whole API)
The tour is **data**, not composed children. Two pieces:
```tsx
const tour = useTour({
  steps,                          // the entire tour — an array of step objects
  closeOnInteractOutside: false,  // default is true; set false so a stray click can't end a guided tour
  onFinish:  () => markTourSeen(),// governance (don't re-show, analytics) lives in your callbacks
  onDismiss: () => markTourSeen(),
})

<Button onClick={() => tour.start()}>Show me around</Button>   {/* launch on demand */}
<Tour tour={tour} />                                           {/* render once, anywhere */}
```
Each **step** is config: `{ id, type, target, title, description, placement, backdrop, effect, actions, meta }`.
`target` is **a function returning the element** (`() => ref.current`), never a selector string.

## Step types
- **`dialog`** — a centered 400px card with **no target**; the welcome/intro step and the celebration/finish step.
- **`tooltip`** — a coachmark **anchored to a `target`** (auto-flips placement); the in-context steps that point at real elements.

## Locked — don't override
- **The sub-components are internal.** `TourContent` / `TourTitle` / `TourFooter` / `TourMedia` etc. are auto-rendered from the step config and **not exported** — never try to compose them. The *only* composable child is `<TourClose>`, and only to attach analytics (it replaces the auto-rendered close).
- **Footer actions are auto-generated by step position** — first: `Skip` + `Start`; middle: `Back` + `Next`; last: `Back` + `Finish`; single-step: `Got it`. Last action is `primary`/`brand`, the rest `ghost` — don't hand-build nav buttons or restyle them. Override the labels/behavior **only** via a step's `actions` array.
- **Spotlight + backdrop are automatic** — the dimmed overlay with a cut-out around the target renders itself; set **`backdrop: false`** per step for a non-blocking hint, and **`meta.shape: 'circle'`** for a round spotlight on an icon-sized target. Don't draw your own overlay.
- **Title clamps to 2 lines, description to 4** (overflow → tooltip). Don't pre-truncate.
- **Keyboard + focus are built in** — arrows move between steps, Esc dismisses, Tab reaches the close button.
- **Progress is automatic** — a step counter renders in the footer whenever the tour has more than one step. Don't build your own "2 of 5" indicator.
- **Steps advance on interaction, not on a timer** — via `Next`, a beacon click, or a `waitForStepEvent` action. There is **no** time-only auto-advance; `waitForStepEvent`'s `delay` is a *debounce on the awaited event* (advance once the input goes quiet), never a standalone countdown.
- **Media is config, not a slot** — set `meta.mediaSrc` (+ `meta.mediaAlt`); image vs video is auto-detected. Don't build your own media area.
- **A step with an `effect` hides the footer's nav buttons entirely** — the user's interaction (or a beacon click) *is* the advance; the only exits are the close (X) and Esc. Don't add a Next button to an effect step.
- **Analytics ride the real nodes** — put `data-analytics-id` / `data-analytics-props` on each step's `actions` entries (they forward to the actual `<button>`) and on `<TourClose>`. No analytics-specific Tour props.

## Sizing / judgment calls
- **`type` per step** — a `dialog` to open (and to celebrate at the end); `tooltip` for everything anchored to a real element.
- **`effect` — the step's mode:**
  - *none* → standard read-and-Next coachmark.
  - **`beaconStepEffect()`** → passive discovery: a pulsing dot appears, the popover opens when the user **clicks** the highlighted element (click, not hover — Tour has no hover-triggered mode; a hover hint is a `Tooltip`). Best for **announcing a new feature without interrupting** (pair with `backdrop: false`).
  - **`waitForStepEvent('click' | 'input' | 'change', args, { predicate, delay })`** → interactive "do it": the step pauses until the user actually performs the action, then auto-advances. **Prefer this on any action-bearing step** — hands-on steps complete and convert far better than passive Next steps.
- **`backdrop: false`** — for subtle, non-blocking hints that keep the rest of the UI live.
- **`closeOnInteractOutside`** (default `true`) — **set `false`** for a deliberate guided tour, so a stray outside click can't end it mid-flow.
- **Launch: on-demand beats blanket.** Prefer a user-initiated trigger (a "Show me around" button, a help-menu entry) or a contextual moment over `autoStart` on first load; contextual tours are far more effective and less resented. If you do `autoStart`, keep it skippable.
- **Keep it short** — 3–5 steps, one goal per tour. The `steps` array is open-ended; brevity is your call, and it's the single biggest lever on completion.
- **`placement`** per `tooltip` step (auto-flips at the viewport edge).

## Writing the steps (house content rules — WIP with technical writers, not frozen)
- **Title: 3–5 words (max 7), benefit-first, sentence case, no period.** Frame the gain, not the feature name: "Track your progress", "Find threats faster", "Your security overview" — not "Progress tracker". (Exclamation marks only on a celebration step.)
- **Description: ≤160 chars (~80–120 ideal), 1–2 sentences.** Lead sentence = what it does / what's new; optional second = why it matters or how to use it. Active voice, address the user as "you / your".
- **One idea per step** — a second idea is a second step. Lead with value, mechanics second. **Never describe what the user can already see** (don't restate a button's label), and only reference **on-screen** elements.
- **Avoid** marketing fluff and superlatives ("amazing", "powerful") and empty adverbs ("easily", "simply"). Technical terms are fine when the audience knows them (Wallarm users know "API endpoint") — just don't stack several in one sentence.
- **Action labels: consistent across the whole tour, and specific.** Never "OK" (ambiguous) — use "Got it" / "Finish" / a real CTA. **Skip is always available; never force completion** — forced tours breed resentment. On a `waitForStepEvent` step, the prompt ("Click **Add rule** to continue") is the description, **not** a button.
- **Final step** — confirm completion ("You're all set") and end on `Finish` or a forward CTA ("Try it now").

## What the component does NOT own (you / the onboarding skill / the app do)
The component renders a tour; it does **not** govern *whether, when, or how often* it runs. You own:
- **Trigger timing** — contextual / user-initiated, not blanket-on-load.
- **Showing it once** — frequency-cap and don't re-show after `onFinish` / `onDismiss` (persist a "seen" flag yourself).
- **The step-count cap and "one highlight on screen at a time"** — the API won't stop you; that's authoring discipline.
- **Success measurement** — completion, step-level drop-off, and downstream activation, via the callbacks + `data-analytics-*`.

## Pairs with
- `useTour()` + a single `<Tour tour={tour} />` — the hook / render pair (the whole public surface).
- `Button` — the launch trigger (`tour.start()`); the footer and close buttons are auto-rendered.
- `beaconStepEffect` / `waitForStepEvent` — the two exported step-effect helpers (passive vs interactive).
- `Badge` (`New`) — marks the entry point that launches or announces the tour.
- `Tooltip` / `Popover` / `Banner` / `Toast` / `Dialog` / `EmptyState` — the alternatives for the cases under "Don't use it for".
